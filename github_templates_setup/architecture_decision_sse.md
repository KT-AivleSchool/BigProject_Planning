## 1. 기술적 문제 정의 (Context)
*   **지연 시간 문제**: GPT-4o 기반 LangGraph 에이전트 간의 3자 토론은 RAG 문서 조회 및 LLM 다단계 추론 프로세스를 거치므로, 최종 타결 시나리오 도출까지 **최소 15초에서 최대 30초 이상의 오랜 처리 지연(Latency)**이 발생합니다.
*   **사용자 경험(UX) 훼손**: 기존의 일반적인 HTTP Request-Response 구조에서는 백엔드 처리가 완전히 끝날 때까지 브라우저 화면이 멈춘(Freeze) 상태가 되어, 공무원(사용자)이 시스템 오작동이나 랙으로 인지하고 브라우저를 이탈하는 치명적인 UX 훼손이 발생합니다.
*   **시연 연출의 필요성**: 심사위원 대상 데모 시, AI 에이전트들이 실시간으로 의견을 주고받는 대화형 행정 시뮬레이션 과정을 시각적으로 매끄럽게 전달할 수 있어야 합니다.

| 평가 기준 | 대안 1. 단방향 폴링 (Short Polling) | 대안 2. 양방향 웹소켓 (WebSocket) | 대안 3. SSE (Server-Sent Events) - 채택 |
| :--- | :--- | :--- | :--- |

| **연결 비용** | 🔴 **최악** (매번 3-Way Handshake 및 헤더 오버헤드) | 🟡 **높음** (프로토콜 변환 ws:// 및 연결 생명주기 관리 필요) | 🟢 **최소** (표준 HTTP 연결을 스트리밍 모드로 재활용) |

| **적합한 유스케이스** | 갱신 주기가 10초 이상인 느린 데이터 | 실시간 대화형 멀티 채팅, 온라인 게임 | **ChatGPT 식의 실시간 텍스트 타이핑 스트리밍** |

## 3. 최종 결정 사유 (Decision Rationale)

### ➊ 요구사항의 단방향 일치성 (One-way Push)
*   모의 심의 기능은 사용자가 토론 중간에 말을 끼워 넣는 채팅이 아니라, **"백엔드 AI 엔진이 작성하는 모의 심의 중계 로그를 사용자가 모니터링하여 감상하는 구조"**입니다. 
*   따라서 무겁고 설계가 복잡한 양대 소켓(WebSocket) 대신, 단방향 실시간 스트리밍에 최적화된 SSE가 구조적으로 완벽히 부합합니다.

### ➋ 클라우드 인프라 배포의 안전성 (Cloud Deployment Security)
*   **AWS EKS 및 API Gateway**를 통해 배포할 때, 웹소켓(`ws://`)은 HTTP 헤더 스왑 설정 및 인그레스(Ingress) 단에서의 지속 유지 세션 튜닝이 까다로우며 무중단 배포 시 연결이 튀어 끊기는 에러가 잦습니다.
*   반면 SSE는 일반 HTTPS 표준 포트(443) 위에서 작동하므로 별도의 인프라 튜닝 없이 **보안 게이트웨이를 그대로 100% 무결하게 통과**합니다.

### ➌ 브라우저의 기본 재연결 메커니즘 (Built-in Auto Reconnect)
*   네트워크의 순간적 불안정으로 통신이 잠시 유실되더라도, 브라우저 내장 `EventSource` API가 기본적으로 백엔드를 향해 자동 재연결(Auto-Reconnect)을 수행하므로 비정상 종료 처리 복구에 강력합니다.

## 4. 기대 효과 (Expected Outcomes)
1.  **시연 극대화**: 심사위원 데모 시 한 글자씩 실시간으로 작성되는 연막 소독, 운영시간 제한 등 타결 협상 내용의 실시간 타이핑 중계 화면을 구축하여 기술적 생동감을 극대화합니다.
2.  **API 비용 최소화**: 스트리밍 통신이 열려 있는 동안 브라우저가 화면을 선점하게 하고 완료 직후 즉시 세션을 종료함으로써 클라우드 네트워크 트래픽 오버헤드를 최소한으로 절감합니다.
